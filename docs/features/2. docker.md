# Déploiement Docker

## Résumé

Les 5 microservices sont packagés et orchestrés via Docker : un `Dockerfile` par service + un `docker-compose.yml` unique à la racine. Objectif : un évaluateur disposant seulement de Docker Desktop peut lancer l'application complète en deux commandes, sans installer Java, Maven, PostgreSQL ni MongoDB.

## Comment lancer le projet

```
./scripts/generate-keys.sh   # génère la paire de clés RS256 (prérequis, voir 3. security.md)
docker compose up --build
```

Puis se rendre sur `http://localhost:8082`. Premier build : plusieurs minutes (téléchargement des dépendances Maven et des images de base pour les 5 modules) ; les lancements suivants réutilisent le cache Docker et sont nettement plus rapides.

## Dockerfiles (un par service)

- **Multi-stage** : un stage `maven:3.9-eclipse-temurin-21` pour le build, un stage `eclipse-temurin:21-jre` pour le runtime — l'image finale ne contient ni JDK ni Maven, seulement le jar et un JRE.
- **Contexte de build = racine du repo**, pas le dossier du module : le mono-repo Maven impose que le pom parent et les 5 poms enfants soient présents pour que le reactor démarre. Chaque `Dockerfile` copie donc les 5 poms avant les sources, puis construit uniquement le module ciblé (`mvn -pl <module> -am`).
- **Ordre des `COPY` optimisé pour le cache Docker** : poms d'abord (résolution des dépendances), sources ensuite — tant qu'aucun pom ne change, la couche de dépendances (longue à construire) reste en cache même quand le code applicatif change.
- **Tests exclus du build d'image** (`-DskipTests`) : ils tournent en local/CI, pas à chaque `docker build` (les tests Testcontainers exigeraient sinon un Docker-dans-Docker).
- Les clés RS256 sont embarquées dans le jar au build (`classpath:keys/`) : rien à monter en volume.

## docker-compose.yml (racine)

7 conteneurs : les 5 services applicatifs + `postgres:16` + `mongo:7`.

- Les noms de services (`patient-service`, `notes-service`...) servent de hostnames DNS internes : la communication inter-services passe par ces noms (`http://patient-service:8081`), jamais par `localhost`.
- **Un seul port publié sur l'hôte : `8082`** (frontend). Le reste (gateway, backs, bases) ne vit que sur le réseau Docker interne (`medilabo-net`) — voir `1. architecture.md`.
- Variables d'environnement avec défauts fonctionnels (`${VAR:-defaut}`) : `docker compose up --build` fonctionne sans créer de `.env`, tout en restant personnalisable via un `.env` à la racine.
- Volumes nommés (`postgres-data`, `mongo-data`) : les données persistent entre redémarrages.

## Healthchecks & ordre de démarrage

`depends_on` seul ne garantit que l'ordre de *démarrage des conteneurs*, pas la *disponibilité* du service qu'ils contiennent — un back qui démarre avant que sa base accepte les connexions crasherait.

- **Bases** : healthchecks natifs (`pg_isready` pour Postgres, `mongosh --eval "db.adminCommand('ping')"` pour Mongo).
- **Applications** : `curl -f http://localhost:<port>/actuator/health` — Spring Boot Actuator, exposant uniquement `health`, ajouté aux 5 services pour cet usage.
- **`depends_on: condition: service_healthy`** en cascade : bases → patient/notes → assessment → gateway → frontend. Chaque niveau n'attend le suivant qu'une fois réellement prêt, garantissant un démarrage ordonné sans crash au premier `up`.

## Décisions & hypothèses

- **Bases conteneurisées, une par service** (aucune installation locale de SGBD à prévoir) — cohérent avec le principe « un processus par conteneur » et la persistance par service (voir `1. architecture.md`).
- **Un seul port publié** : choix de sécurité autant que de configuration — le réseau Docker matérialise la contrainte « la gateway est l'unique point d'entrée ».
- **Défauts fonctionnels dans le compose** : priorité à la simplicité de démarrage pour l'évaluateur ; les vrais secrets restent personnalisables via un `.env` racine, jamais commité.

## Limitations connues

- Premier `docker compose up --build` relativement long (dépendances Maven + images de base pour 5 modules) ; les suivants sont mis en cache.
- Pas d'authentification sur MongoDB en démo — cohérent avec un contexte local, pas un déploiement de production.
- `scripts/generate-keys.sh` reste une étape manuelle avant le premier lancement (les clés RS256 ne sont jamais commises, même de démonstration) : deux commandes plutôt qu'une seule, assumé.
