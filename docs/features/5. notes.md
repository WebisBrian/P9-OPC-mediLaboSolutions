# Notes médecin

## Résumé

Permet à un médecin de consulter l'historique des notes d'un patient et d'en ajouter de nouvelles. Les notes sont du texte libre, non structuré — c'est la donnée que l'algorithme d'assessment analyse pour détecter des facteurs de risque.

## Endpoints (notes-service, via la gateway)

| Méthode | Chemin | Description |
|---|---|---|
| GET | `/notes?patientId={id}&page={p}&size={s}` | Historique paginé des notes d'un patient (tri décroissant par date) |
| POST | `/notes` | Ajout d'une note |

Pas de GET par id, pas de PUT/DELETE : seules ces deux opérations sont demandées par les user stories.

## Architecture impactée

- **notes-service** (port 8083) — persistance MongoDB (collection `notes`), aucune relation SQL avec patient-service.
- **gateway-service** — route `/notes/**`.
- **frontend-service** — section "Historique des notes" + formulaire d'ajout sur la page détail patient (`PatientController`, `NoteGatewayClient`), pagination Bootstrap.

## Décisions & hypothèses

- `patientId` est une simple référence (`Long`) : aucune donnée patient n'est dupliquée dans la note, la source de vérité reste patient-service.
- MongoDB plutôt que PostgreSQL : le contenu est un texte libre sans structure fixe, sans jointure nécessaire (voir `1. architecture.md` pour la logique de persistance polyglotte).
- `createdAt` (horodatage de création, non modifiable après coup) a été ajouté bien que non demandé explicitement — déduit du besoin "consulter l'historique", qui implique une chronologie.
- Aucune vérification que `patientId` correspond à un patient existant au moment de la création (pas d'appel réseau vers patient-service) — cohérent avec l'autonomie des services.

## Comment tester

- Tests unitaires (service, mapper) et `@WebMvcTest` (controller, sécurité exclue) dans `notes-service`.
- Tests d'intégration base de données sur Testcontainers (`mongo:7`).
- Manuel : page détail d'un patient, ajouter une note, vérifier son apparition dans l'historique et la pagination.

## Limitations connues

- Une note peut référencer un `patientId` inexistant (pas de contrainte d'intégrité inter-service) : sans conséquence pratique, l'assessment part toujours du patient et échoue avant même de lire les notes si celui-ci n'existe pas.
- Pas de suppression en cascade des notes si le patient associé est supprimé.
- Horodatages affichés en UTC, sans conversion de fuseau.
