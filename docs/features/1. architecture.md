# Architecture microservices

## Vue d'ensemble

5 microservices Spring Boot, déployés ensemble via un `docker-compose.yml` unique à la racine du repo.

| Service | Rôle | Port | Persistance |
|---|---|---|---|
| gateway-service | Point d'entrée unique, sécurité | 8080 | — |
| patient-service | CRUD patient | 8081 | PostgreSQL |
| frontend-service | Interface utilisateur (Thymeleaf) | 8082 | — |
| notes-service | Notes médecin | 8083 | MongoDB |
| assessment-service | Calcul du risque diabète | 8084 | — (interroge patient + notes) |

## La gateway comme point d'entrée unique

Spring Cloud Gateway (WebFlux, réactif — adapté à un proxy à forte concurrence, non bloquant) route chaque requête entrante par préfixe de chemin vers le back concerné (`/patients/**`, `/notes/**`, `/assessment/**`). C'est aussi elle qui authentifie l'utilisateur (voir `3. security.md`) : aucun autre service ne gère d'authentification utilisateur.

En Docker, un seul port est publié sur l'hôte : `8082` (le frontend). Ni la gateway, ni les backs, ni les bases ne sont accessibles depuis l'extérieur — ils vivent uniquement sur le réseau Docker interne (`medilabo-net`). Le rôle de point d'entrée unique de la gateway est ainsi garanti par le réseau, pas seulement par convention applicative.

## Communication interne : trafic entrant vs trafic interne

- **Trafic entrant** (client → système) : passe systématiquement par la gateway — c'est son rôle.
- **Trafic interne** (service → service) : `assessment-service` appelle `patient-service` et `notes-service` **directement**, sans repasser par la gateway. Router ce trafic interne par la gateway ajouterait un aller-retour réseau inutile et en ferait un point de passage obligé même pour des appels purement internes au système.

## Persistance par service

Chaque service qui persiste des données a sa **propre base**, jamais partagée entre services :

- `patient-service` → PostgreSQL — données structurées, relationnelles.
- `notes-service` → MongoDB — texte libre non structuré, pas de schéma rigide à imposer, pas de jointure nécessaire.
- `assessment-service` n'a aucune base : il recalcule le risque à chaque appel à partir des deux autres services.

Conséquence assumée : aucune contrainte de clé étrangère n'est possible entre services (ex. le `patientId` d'une note n'est jamais vérifié en base contre `patient-service`) — c'est le prix de l'indépendance de déploiement propre aux microservices, pas un oubli.

## Le frontend

Spring MVC + Thymeleaf, rendu côté serveur (pas de SPA, pas de build JavaScript). Toujours indépendant des backs : il ne leur parle jamais directement, uniquement à la gateway, via un client REST (`RestClient`) dédié par domaine (`PatientGatewayClient`, `NoteGatewayClient`, `AssessmentGatewayClient`).

## Déploiement

Les services tournent en conteneurs Docker, orchestrés par un unique `docker-compose.yml` à la racine. Détail des Dockerfiles, du compose, des healthchecks et de la commande de lancement : `2. docker.md`.

## Limitations connues

- Pas de haute disponibilité : la gateway est un point de défaillance unique (cohérent avec l'échelle du projet).
- Pas de service discovery : les URLs des services sont en configuration statique (noms Docker Compose), pas de registre dynamique.
- Pas de circuit breaker sur les appels inter-services (assessment → patient/notes) : une panne d'un service dépendant est gérée au cas par cas par l'appelant, pas par un mécanisme générique.
