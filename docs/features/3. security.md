# Sécurité

## Résumé

Trois segments de communication, trois mécanismes distincts, chacun adapté à la nature de la liaison qu'il protège.

| Segment | Mécanisme |
|---|---|
| Navigateur ↔ frontend | Session serveur (cookie `JSESSIONID`) |
| Frontend ↔ gateway | HTTP Basic (credentials utilisateur rejoués depuis la session) |
| Gateway ↔ services back | JWT RS256 signé (`Authorization: Bearer`) |

## Authentification utilisateur (frontend + gateway)

- Le frontend n'a pas de Spring Security : c'est un login "maison" léger. La page de login teste les identifiants en appelant la gateway ; s'ils sont valides, ils sont stockés dans la `HttpSession` puis rejoués en HTTP Basic à chaque appel gateway. Un `HandlerInterceptor` protège les routes nécessitant une session active.
- La gateway (Spring Cloud Gateway, WebFlux) authentifie réellement l'utilisateur : un seul utilisateur in-memory, mot de passe hashé BCrypt, HTTP Basic (`SecurityWebFilterChain`).
- Aucune inscription, aucun rôle ni droit — hors périmètre du projet.

## Propagation d'identité interne (JWT RS256)

La communication gateway → back (et back → back pour assessment) repose sur un JWT signé plutôt que sur un identifiant statique : un token prouve à la fois la provenance de la requête (elle vient bien de la gateway) et l'identité de l'utilisateur (`sub`), en un seul mécanisme cryptographique.

- **Émission (gateway)** — après authentification Basic réussie, `JwtIssuer` forge un JWT signé avec la clé **privée** RS256 de la gateway. Claims : `sub` (username), `iss` (`medilabo-gateway`), `iat`, `exp` — expiration courte (60 secondes, configurable), car le token ne sert qu'à un seul saut réseau interne. `JwtRelayGlobalFilter` l'injecte en `Authorization: Bearer` sur chaque requête sortante, en **remplaçant** le header Basic entrant : les credentials utilisateur ne franchissent jamais la gateway vers les backs.
- **Validation (chaque back)** — `spring-boot-starter-oauth2-resource-server` standard (pas de filtre de validation maison). Vérifie signature RS256 (avec la clé **publique**), expiration, et issuer. Sans token valide → `401`.
- **Relais (assessment-service)** — seul service à la fois serveur et client. Il ne forge jamais de token : il relaie tel quel celui qu'il a reçu (`RelayedJwtProvider`) vers ses appels sortants à patient-service et notes-service. L'identité de l'utilisateur d'origine traverse ainsi toute la chaîne d'agrégation.

## Pourquoi RS256 (asymétrique) plutôt qu'un secret partagé symétrique

Seule la gateway détient la clé privée : aucun back ne peut forger de token, même compromis. Avec une clé symétrique partagée entre tous les services, n'importe lequel d'entre eux pourrait signer un faux token au nom de n'importe quel utilisateur.

## Clés

- Paire RS256 générée par `scripts/generate-keys.sh` (clé privée PKCS#8 PEM côté gateway, clé publique X.509/SPKI PEM dupliquée dans chaque back).
- Clés de production non versionnées (gitignore). Clés de test jetables committées dans `src/test/resources/keys/` de chaque service, nécessaires pour que les tests `@SpringBootTest` passent en CI.

## Secrets & configuration

- Credentials (gateway HTTP Basic, DB Postgres/Mongo) via variables d'environnement / `.env` — jamais en dur dans le code.
- Les chemins de clés (pas leur contenu) ont un défaut dans la configuration (`classpath:keys/...`) : ce ne sont pas des secrets.
- Le frontend n'a pas de `.env` : pas de compte de service, les credentials viennent de l'utilisateur via la page de login.

## Limitations connues

- Un seul utilisateur, aucun rôle/droit exprimé dans le JWT (`sub` seulement).
- HTTP Basic non chiffré sur le segment frontend ↔ gateway : sans HTTPS devant le frontend, les credentials transitent en clair sur le réseau (le Base64 de HTTP Basic est un encodage, pas un chiffrement). Hors périmètre du projet (pas de TLS demandé).
- Pas de rotation de clés, pas de JWKS, pas de révocation de token, pas de refresh token : le périmètre est volontairement limité à émettre / signer / valider.
