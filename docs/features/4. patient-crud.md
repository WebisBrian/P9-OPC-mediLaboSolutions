# CRUD Patient

## Résumé

Gestion des données démographiques d'un patient : création, consultation, modification, suppression. C'est la fonctionnalité socle du projet — les notes et l'évaluation du risque s'appuient sur l'existence d'un patient.

## Endpoints (patient-service, via la gateway)

| Méthode | Chemin | Description |
|---|---|---|
| GET | `/patients` | Liste tous les patients |
| GET | `/patients/{id}` | Détail d'un patient |
| POST | `/patients` | Création |
| PUT | `/patients/{id}` | Modification |
| DELETE | `/patients/{id}` | Suppression |

Champs : `firstName`, `lastName`, `birthDate`, `gender` (`M`/`F`), `address`, `phone` (les deux derniers optionnels).

## Architecture impactée

- **patient-service** (port 8081) — CRUD REST, persistance PostgreSQL via JPA. Couches `controller` / `service` / `repository` / `mapper` / `dto` / `exception`.
- **gateway-service** — route `/patients/**` vers patient-service.
- **frontend-service** — `PatientController` + `PatientGatewayClient`, pages Thymeleaf (liste, détail, formulaire création/édition).

## Décisions & hypothèses

- Validation dupliquée côté back (rempart d'intégrité) et côté front (confort UX) — mêmes règles, sans code partagé entre les deux services.
- Détection de doublon métier (même nom/prénom/date de naissance/téléphone) → `409 Conflict`.
- `gender` est un enum persisté en base sous forme de texte lisible (`M`/`F`), mais manipulé comme une simple `String` côté front pour éviter toute dépendance à un type interne du back.
- Le champ JSON s'appelle `birthDate` (D majuscule) — contrat à respecter par tout client de l'API.

## Comment tester

- Tests unitaires (service, mapper) et `@WebMvcTest` (controller, sécurité exclue) dans `patient-service`.
- Tests d'intégration base de données sur Testcontainers (`postgres:16`).
- Manuel : `docker compose up --build`, se connecter sur `http://localhost:8082`, créer/modifier/supprimer un patient.

## Limitations connues

Aucune limitation notable — fonctionnalité complète sur son périmètre.
